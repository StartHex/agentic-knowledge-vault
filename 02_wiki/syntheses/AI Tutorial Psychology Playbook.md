## Purpose

This page stores psychology and persuasion principles that can improve AI tutorial selection, scripting, editing, and review. It should be used as a lens for converting collected AI news and tool tutorials into practical creator ideas.

## Core Audience Psychology

Most viewers do not only want "AI news." They want reduced uncertainty:

- What changed?
- Does it affect me?
- Can I use it today?
- Is it worth learning?
- Will I look behind if I ignore it?

An AI tutorial performs well when it turns abstract technology into a concrete reduction of time, cost, risk, or confusion.

## Reusable Principles

### 1. Curiosity Gap

Use a specific missing answer, not vague hype.

Weak:

```text
这个工具太强了。
```

Stronger:

```text
我让 Codex 自己改项目，但它第一次失败在这个配置上。
```

Use for:

- opening hooks
- section transitions
- titles

### 2. Loss Aversion

People react strongly to missed opportunities and avoidable mistakes.

Use carefully:

- "别先装插件，先检查这个环境变量。"
- "很多人看完新闻却没意识到，它真正改变的是工作流。"
- "如果 Obsidian 路径错了，你以为同步了，其实图谱看不到。"

Avoid fear-mongering. Tie the risk to a practical fix.

### 3. Social Proof

Use attention signals as a reason to investigate, not as proof of truth.

Valid signals:

- GitHub stars and recent commits
- high read/like/comment counts
- repeated mentions across multiple sources
- official docs or release notes
- respected creators independently covering the same topic

Script phrasing:

```text
它今天被多个来源同时提到，所以我不先下结论，直接拿一个真实任务测。
```

### 4. Authority With Verification

Borrow authority from official docs, source code, changelogs, and reproducible tests. Do not rely only on big creator claims.

Best pattern:

```text
大号说它很强，我看了官方文档，然后用一个真实任务验证。
```

This fits the account's "AI handbook" position: useful, skeptical, reproducible.

### 5. Cognitive Load Reduction

A tutorial should reduce decisions for the viewer.

Do:

- give one recommended path
- explain optional choices later
- name files and buttons exactly
- show a final verification step

Do not:

- list too many alternatives early
- switch tools mid-tutorial
- explain architecture before the viewer sees why it matters

### 6. Identity And Aspiration

Frame the viewer as someone building an AI operating system, not merely chasing tools.

Useful positioning:

- "把新闻变成自己的 AI 手册"
- "把工具教程变成可复用工作流"
- "从看热闹变成可执行资产"

This supports long-term monetization because the account becomes a system, not a daily hot-take feed.

### 7. Commitment Ladder

Design content so viewers can move from light engagement to deeper trust.

Ladder:

- short clip: one visible trick
- article/thread: full steps
- knowledge base note: durable checklist
- template/package: downloadable asset
- course/consulting: complete system

Every topic should be scored by where it can sit on this ladder.

### 8. Specificity Bias

Specific numbers, paths, commands, and screenshots feel more trustworthy than broad claims.

Prefer:

- "每天 08:00 和 20:00 自动采集"
- "`10.106.106.80:9877` 本地审核面板"
- "`AI采集日报 2026-06-16` 出现在 Obsidian 图谱"

Avoid unsupported performance claims unless measured.

## Applying This To AI Tutorial Selection

When a collected item appears, ask:

- Can it create a strong before/after contrast?
- Can it be verified with a small demo?
- Does it reduce uncertainty for creators, programmers, or operators?
- Does it fit an existing series such as Codex, 国产 Codex, AI 自动化, or AI 视频?
- Can it become a reusable page in the AI handbook?

## Title And Hook Formulas

Use these as drafts, not final copy:

- "我用 X 跑了一个真实任务，结果卡在 Y"
- "别急着学 X，先看它到底替你省了哪一步"
- "大号都在讲 X，我只测一个问题：普通人能不能复现"
- "把 X 接进 Obsidian 后，AI 新闻终于变成知识库"
- "X 不是另一个工具，它改变的是这条工作流"

## Review Checklist

Before choosing a topic:

- Is the psychological promise concrete?
- Is the proof visible?
- Is the difficulty honest?
- Is the next action obvious?
- Can the viewer reuse the result tomorrow?
