# github-devpor 详细设计

## 1. 文档目标

本文档基于 `github-devpor设计.md`，将概念方案细化为可落地的工程设计，覆盖客户端（Electron + React）、服务端（支付与验证）、数据模型、安全策略、关键流程、异常处理与开发里程碑。

目标是支持以下核心能力：
- GitHub 项目搜索、README 预览、国内代理下载
- 普通安装与懒人安装（AI 生成命令并执行）
- 内置模型的按次付费授权（2.99 元/次）
- 按需联网：普通模式和自定义模型模式默认不连接业务服务器

## 2. 总体架构

### 2.1 分层

- `renderer`：UI、交互状态、日志展示
- `preload`：安全桥接，仅暴露白名单 API
- `electron main`：下载、文件、命令执行、代理管理、支付轮询、加密存储
- `server`：订单创建/回调/校验/恢复、动态代理列表
- 第三方：GitHub API、代理源、支付网关、可选 AI API（OpenAI 兼容）

## 3. 模块详细设计

### 3.1 搜索与 README 模块

#### 功能
- 支持 `URL` 与 `关键词` 两种搜索模式
- 关键词搜索可按 star 排序
- 选中项目后加载 README 并在同面板内导航链接

#### 关键点
- 前端输入防抖：`300ms`
- 主进程统一发 GitHub 请求并套代理
- README 渲染：`react-markdown` + 链接点击拦截
- 对 GitHub API 限流/失败进行降级提示

#### IPC
- `search:repos`：入参 `{ type, query, sort }`，返回项目列表
- `repo:getReadme`：入参 `{ owner, repo, ref? }`，返回 markdown 字符串

### 3.2 代理下载模块

#### 功能
- 支持源码与 Release 资产下载
- 自动选择当前平台可执行包
- 下载进度、重试、代理故障转移

#### 核心策略
- 代理池：内置列表 + 可选远程列表
- 健康检查：启动时/每 12 小时测速，缓存可用结果
- 故障切换：按延迟排序逐个尝试，单代理失败重试 3 次

#### IPC
- `download:start`：入参 `{ url, mode, targetPath }`，返回 taskId
- `download:cancel`：入参 `{ taskId }`
- 事件 `download:progress`：`{ taskId, percent, speed, eta }`
- 事件 `download:done` / `download:error`

### 3.3 懒人安装（AI）模块

#### 功能
- 扫描项目关键文件，构造提示词，让 AI 返回安装命令序列
- 命令执行前校验风险，逐条执行并实时回传日志

#### 流程
1. 下载/解压项目到临时目录
2. 扫描 `README.md`、`package.json`、`setup.py`、`pyproject.toml`、`Makefile` 等
3. 根据系统类型拼接 prompt
4. 调用模型（内置/自定义）
5. 命令安全过滤与用户确认
6. 逐条执行，失败时用户可重试/跳过/终止

#### 安全规则
- 禁止：`rm -rf /`、`sudo`、重定向系统目录、危险 PowerShell 脚本
- 单命令超时（默认 10 分钟）、输出上限（默认 1MB）
- 执行目录限定为项目路径

#### IPC
- `install:plan`：返回命令列表与风险标记
- `install:execute`：开始执行
- `install:action`：对失败步骤执行 `retry|skip|abort`
- 事件 `install:log`、`install:step`

### 3.4 支付与授权模块

#### 业务规则
- 使用内置模型的懒人安装：`2.99 元 / 次`
- 使用自定义模型：免费，不触发支付逻辑，不记录付费核验日志

#### 流程
1. 客户端创建订单（带 `deviceId`）
2. 展示二维码并每 2 秒轮询状态
3. 支付成功后调用 `verify/use`
4. 服务端标记订单可用/已用状态
5. 客户端落地本地加密授权记录，扣减一次

#### IPC
- `payment:createOrder`
- `payment:pollStatus`
- `payment:verifyUse`
- `payment:restoreByOrder`

### 3.5 按需联网策略

#### 判定
- 普通模式：不连接业务服务器
- 懒人 + 自定义模型：不连接业务服务器
- 懒人 + 内置模型：连接业务服务器（支付/验证）

#### 实现
- 统一 `connectionManager.isServerRequired(mode, aiProvider)`
- 所有服务端 API 经过统一封装，先检查开关
- 代理远程列表获取为可选优化，不影响基础下载

## 4. 数据设计

### 4.1 客户端本地存储

建议基于 `electron-store`，文件位于 `app.getPath('userData')`，内容加密后写入。

```ts
interface LocalStore {
  deviceId: string;
  settings: {
    mode: "normal" | "lazy";
    aiProvider: "builtin" | "custom";
    aiBaseUrl?: string;
    aiApiKey?: string;
    aiModel?: string;
    proxyMode: "auto" | "custom";
    customProxy?: string;
  };
  orders: Array<{
    orderId: string;
    status: "pending" | "paid" | "used" | "expired";
    createdAt: number;
    usedAt?: number;
  }>;
  logs: Array<{
    id: string;
    orderId?: string;
    encrypted: string;
    createdAt: number;
  }>;
}
```

### 4.2 服务端数据表

`orders`
- `id`（主键）
- `device_id`
- `amount`（分）
- `status`：`pending|paid|used|expired`
- `paid_at`、`used_at`
- `restored_device_id`
- `created_at`

`proxy_list`
- `version`
- `proxies`（json）
- `updated_at`

## 5. 接口设计（服务端）

### 5.1 订单
- `POST /api/order/create`
  - req: `{ deviceId, amount }`
  - resp: `{ orderId, qrcodeUrl, expireTime }`
- `POST /api/order/webhook`
  - 支付回调，验签后更新订单状态
- `GET /api/order/status?orderId=&deviceId=`
  - resp: `{ status }`

### 5.2 使用验证与恢复
- `POST /api/verify/use`
  - req: `{ orderId, deviceId }`
  - resp: `{ valid, remainingUses }`
- `POST /api/restore`
  - req: `{ orderId, newDeviceId }`
  - resp: `{ success, message }`

### 5.3 代理列表
- `GET /api/proxy/list`
  - resp: `{ version, proxies }`

## 6. 状态管理设计（Renderer）

建议 `Zustand` 切分为 4 个 store：
- `useSearchStore`：query、results、selectedRepo、loading
- `useInstallStore`：mode、plan、steps、logs、running
- `usePaymentStore`：remaining、order、paying、status
- `useSettingsStore`：代理、AI 参数、隐私开关

这样可以降低单 store 复杂度，避免无关组件频繁重渲染。

## 7. 安全设计

- 本地加密：AES-256-GCM，密钥由 `deviceId + salt` 派生
- 支付回调必须验签，拒绝未签名请求
- 订单防重放：同一订单同一设备仅允许消费一次
- API 限流：按 IP + deviceId 限制创建订单频率
- 命令执行隔离：限定 cwd、黑名单校验、超时与输出截断
- 隐私策略：自定义模型模式下不保存敏感安装日志

## 8. 可观测性与日志

- 客户端日志等级：`info/warn/error`
- 付费场景下保留最近 3 次加密执行日志（单次 <= 50KB）
- 服务端记录订单流转与 webhook 结果，便于审计与客服排障

## 9. 异常与降级策略

- GitHub API 失败：提示切换关键词、稍后重试或改用 URL 模式
- 代理全部失效：切换到原始链接（可选），并提示网络限制风险
- 支付超时：订单置 `expired`，允许用户重建
- AI 响应异常：允许手动编辑命令后继续执行
- 命令失败：提供重试、跳过、终止三种操作

## 12. 可执行任务清单（Backlog）

本清单按 `M1 -> M4` 顺序执行。每个任务包含：`任务内容`、`交付物`、`验收标准`、`前置依赖`。建议用任务编号直接创建 issue。

- 交付物：可运行工程、基础目录、`npm run dev` 联调脚本。
- 验收标准：开发模式可启动；渲染进程可通过 preload 调用一个测试 IPC（如 `app:getVersion`）。

#### T1-02 IPC 通道与类型定义
- 任务内容：定义 IPC 命名规范、请求/响应 TypeScript 类型、错误码结构。

- 验收标准：所有已实现通道可类型检查通过；前后端无 `any` 泄漏。
- 前置依赖：T1-01。

#### T1-03 搜索模块（URL/关键词）
- 任务内容：实现 `search:repos`，支持 URL 解析和 GitHub 关键词搜索。
- 交付物：主进程搜索服务、前端搜索栏与结果列表。

#### T1-04 README 加载与渲染

#### T1-05 普通下载与解压

- 验收标准：一条完整链路可稳定执行 10 次以上无阻塞性错误。
- 前置依赖：T1-03、T1-04、T1-05、T1-06。

### 12.5 任务优先级与并行建议

- P0（必须优先）：T1-01~T1-07、T2-03、T2-04、T3-01、T3-03、T3-05、T4-05

- P2（优化项）：代理远程列表高级策略、客服日志导出增强、更多 AI 提示模板

- 稳定性主线：M4 代理治理 + 错误治理

### 12.6 Definition of Done（统一完成标准）
